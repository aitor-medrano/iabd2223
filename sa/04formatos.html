    Formatos de datos en Big Data

    En la primera unidad de trabajo ya vimos una peque√±a introducci√≥n a los diferentes formatos de datos.

    Las propiedades que ha de tener un formato de datos son:

    • independiente del lenguaje
    • expresivo, con soporte para estructuras complejas y anidadas
    • eficiente, r√°pido y reducido
    • din√°mico, de manera que los programas puedan procesar y definir nuevos tipos de datos.
    • formato de fichero standalone y que permita dividirlo y comprimirlo.

    Para que Hadoop/Spark o cualquier herramienta de anal√≠tica de datos pueda procesar documentos, es imprescindible que el formato del fichero permita su divisi√≥n en fragmentos (splittable in chunks).

    Si los clasificamos respecto al formato de almacenamiento tenemos:

    • texto (m√°s lentos, ocupan m√°s pero son m√°s expresivos y permiten su interoperabilidad): CSV, XML, JSON, etc...
    • binarios (mejor rendimiento, ocupan menos, menos expresivos): Avro, Parquet, ORC, etc...

    Si comparamos los formatos m√°s empleados a partir de las propiedades descritas tenemos:

    Caracter√≠stica CSV XML / JSON SequenceFile Avro
    Independencia del lenguaje
    Expresivo
    Eficiente
    Din√°mico
    Standalone
    Divisible

    Las ventajas de elegir el formato correcto son:

    • Mayor rendimiento en la lectura y/o escritura
    • Ficheros troceables (splittables)
    • Soporte para esquemas que evolucionan
    • Soporte para compresi√≥n de los datos (por ejemplo, mediante Snappy).

    Filas vs Columnas¶

    Los formatos con los que estamos m√°s familiarizados, como son CSV o JSON, se basan en filas, donde cada registro se almacena en una fila o documento. Estos formatos son m√°s lentos en ciertas consultas y su almacenamiento no es √≥ptimo.

    En un formato basado en columnas, cada fila almacena toda la informaci√≥n de una columna. Al basarse en columnas, ofrece mejor rendimiento para consultas de determinadas columnas y/o agregaciones, y el almacenamiento es m√°s √≥ptimo (como todos los datos de una columna son del mismo tipo, la compresi√≥n es mayor).

    Avro¶

    Apache Avro es un formato de almacenamiento basado en filas para Hadoop, utilizado para la serializaci√≥n de datos, ya que es m√°s r√°pido y ocupa menos espacio que JSON, debido a que la serializaci√≥n de los datos se realiza en un formato binario compacto.

    Tiene soporte para la compresi√≥n de bloques y es un formato que permite la divisi√≥n de los datos (splittable).

    Formato¶

    El formato Avro se basa en el uso de esquemas, los cuales definen los tipos de datos y protocolos mediante JSON. Cuando los datos .avro son le√≠dos siempre est√° presente el esquema con el que han sido escritos.

    Cada fichero Avro almacena el esquema en la cabecera del fichero y luego est√°n los datos en formato binario.

    Los esquemas se componen de tipos primitivos (null, boolean, int, long, float, double, bytes, y string) y compuestos (record, enum, array, map, union, y fixed).

    Un ejemplo de esquema podr√≠a ser:

    empleado.avsc
    {
   "type" : "record",
   "namespace" : "SeveroOchoa",
   "name" : "Empleado",
   "fields" : [
      { "name" : "Nombre" , "type" : "string" },
      { "name" : "Altura" , "type" : "float" }
      { "name" : "Edad" , "type" : "int" }
   ]
}

    Avro y Python¶

    Para poder serializar y deserializar documentos Avro mediante Python, previamente debemos instalar la librer√≠a avro:

    pip install avro-python3
# o si utilizamos Anaconda

    Vamos a realizar un ejemplo donde primero leemos un esquema de un archivo Avro, y con dicho esquema, escribiremos nuevos datos en un fichero. A continuaci√≥n, abrimos el fichero escrito y leemos y mostramos los datos:

    import avro
import copy
import json
from avro.datafile import DataFileReader, DataFileWriter
from avro.io import DatumReader, DatumWriter

# abrimos el fichero en modo binario y leemos el esquema

# escribimos un fichero a partir del esquema le√≠do
with open('empleados.avro', 'wb') as f:
    writer = DataFileWriter(f, DatumWriter(), schema)
    writer.append({"nombre": "Carlos", "altura": 180, "edad": 44})
    writer.append({"nombre": "Juan", "altura": 175})
    writer.close()

# abrimos el archivo creado, lo leemos y mostramos l√≠nea a l√≠nea
with open("empleados.avro", "rb") as f:
    reader = DataFileReader(f, DatumReader())
    # copiamos los metadatos del fichero le√≠do
    metadata = copy.deepcopy(reader.meta)
    # obtenemos el schema del fichero le√≠do
    schemaFromFile = json.loads(metadata['avro.schema'])
    # recuperamos los empleados
    empleados = [empleado for empleado in reader]
    reader.close()

print(f'Schema de empleado.avsc:\n {schema}')
print(f'Schema del fichero empleados.avro:\n {schemaFromFile}')
print(f'Empleados:\n {empleados}')

    Acceso HDFS

    df = pd.read_csv(reader,sep=';')

# Limpiamos los datos (strip a los c√≥digos postales) y nos quedamos con Alemania
df['Zip'] = df['Zip'].str.strip()
filtro = df.Country=="Germany"
df = df[filtro]

# 3. Definimos el esquema
schema = {
    'name': 'Sales',
    'namespace' : 'SeveroOchoa',
    'type': 'record',
    'fields': [
        {'name': 'ProductID', 'type': 'int'},
        {'name': 'Date', 'type': 'string'},
        {'name': 'Zip', 'type': 'string'},
        {'name': 'Units', 'type': 'int'},
        {'name': 'Revenue', 'type': 'float'},
        {'name': 'Country', 'type': 'string'}
    ]
}

    Parquet¶

    Apache Parquet es un formato de almacenamiento basado en columnas para Hadoop, con soporte para todos los frameworks de procesamiento de datos, as√≠ como lenguajes de programaci√≥n. De la misma forma que Avro, se trata de un formato de datos auto-descriptivo, de manera que embebe el esquema o estructura de los datos con los propios datos en s√≠. Parquet es id√≥neo para analizar datasets que contienen muchas columnas, es decir, para lecturas de grandes cargas de trabajo.

    Tiene un ratio de compresi√≥n muy alto (mediante Snappy ronda el 75%), y adem√°s, solo se recorren las columnas necesarias en cada lectura, lo que reduce las operaciones de disco de entrada/salida.

    Formato¶

    Cada fichero Parquet almacena los datos en binario organizados en grupos de filas. Para cada grupo de filas (row group), los valores de los datos se organizan en columnas, lo que facilita la compresi√≥n a nivel de columna.

    La columna de metadatos de un fichero Parquet se almacena al final del fichero, lo que permite que las escrituras sean r√°pidas con una √∫nica pasada. Los metadatos pueden incluir informaci√≥n como los tipos de datos, esquemas de codificaci√≥n/compresi√≥n, estad√≠sticas, nombre de los elementos, etc...

    Parquet y Python¶

    Para interactuar con el formato Parquet mediante Python, la librer√≠a m√°s utilizada es la que ofrece Apache Arrow, en concreto la librer√≠a PyArrow.

    As√≠ pues, la instalamos mediante pip:

    pip install pyarrow

    Apache Arrow usa un tipo de estructura denominada tabla para almacenar los datos bidimensional (ser√≠a muy similar a un Dataframe de Pandas). La documentaci√≥n de PyArrow dispone de un libro de recetas con ejemplos con c√≥digo para los diferentes casos de uso que se nos puedan plantear.

    Vamos a simular el mismo ejemplo que hemos realizado previamente mediante Avro, y vamos a crear un fichero en formato JSON con empleados, y tras persistirlo en formato Parquet, lo vamos a recuperar:

    dict-parquet.py

                    ('altura', pa.int32()),
                    ('edad', pa.int32())  ])

            "altura": [180, 44],

# 4.- Leemos el fichero generado

schemaFromFile = table2.schema

print(f'Schema del fichero empleados.parquet:\n{schemaFromFile}\n')
print(f'Tabla de Empleados:\n{table2}')

    ORC¶

    Apache ORC es un formato de datos columnar optimizado para la lectura, escritura y procesamiento de datos en Hive. ORC tiene una tasa de compresi√≥n alta (utiliza zlib), y al basarse en Hive, soporta sus tipos de datos (datetime, decimal, y tipos complejos como struct, list, map y union) y es compatible con HiveQL.

    Los fichero ORC se componen de tiras de datos (stripes), donde cada tira contiene un √≠ndice, los datos de la fila y un pie (con estad√≠sticas como la cantidad, m√°ximos y m√≠nimos y la suma de cada columna convenientemente cacheadas)

    ORC y Python¶

    Para crear archivos ORC y leerlos, volvemos a necesitar la librer√≠a PyArrow.

    Comparando formatos¶

    Acabamos de ver que cada uno de los formatos tiene sus puntos fuertes.
